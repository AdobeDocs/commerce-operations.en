# Understand patch-status results

Use this topic to interpret patch-status reports generated by [!DNL Core Version Tool] ([!DNL CVT]). The report helps you identify which monthly security patches are detected, which patches are missing, and whether patch coverage protects specific Common Vulnerabilities and Exposures (CVE) records.

## How [!DNL CVT] determines results

[!DNL CVT] is informational and query-only. It does not apply patches, revert patches, or modify Adobe Commerce source files.

To determine patch status, [!DNL CVT]:

- Reads `composer.lock` to identify the Adobe Commerce base version and installed components.
- Loads patch metadata from `data/patch-registry.json`.
- Tests applicable patch diffs in a temporary working directory.
- Maps detected patch coverage to patch and CVE status values.
- Outputs results in JSON by default or CSV for integrations.

## Patch status values

The patch-status report uses the following patch result values:

| Patch status | Meaning |
| --- | --- |
| Applied | [!DNL CVT] detects the monthly security patch in the Adobe Commerce codebase. |
| Missing | The patch applies to the installed Adobe Commerce version or component, but [!DNL CVT] does not detect it. |
| Unknown | [!DNL CVT] cannot confirm the patch status from the available registry, patch diff, or detection result. |

{style="table-layout:auto"}

Review missing and unknown patches before making security-status claims.

## CVE status values

[!DNL CVT] maps patch coverage to CVE status values so you can understand security exposure.

| CVE status | Meaning |
| --- | --- |
| Protected | The applicable patch is detected for the CVE or component. |
| Vulnerable | An applicable patch is missing for the CVE or component. |
| Not applicable | The CVE applies to a component that is not installed, such as Adobe Commerce business-to-business (B2B) or Adobe Commerce Page Builder. |
| Unknown | [!DNL CVT] cannot determine the CVE status from the available registry and detection data. |

{style="table-layout:auto"}

## Key output fields

The report can include the following information:

- **Base Adobe Commerce version** - The installed base version detected from `composer.lock`.
- **Registry source** - Whether patch metadata came from a remote registry or local cache.
- **Installed components** - Adobe Commerce package areas detected from `composer.lock`.
- **Applied patches** - Monthly security patches detected in the Adobe Commerce codebase.
- **Missing patches** - Monthly security patches that apply but are not detected.
- **Unknown patches** - Patches that [!DNL CVT] cannot classify.
- **Vulnerability status by CVE** - CVE coverage mapped to protected, vulnerable, not applicable, or unknown status.
- **Warnings** - Conditions that might affect the reliability or completeness of the report.

## Patch registry and cache

`data/patch-registry.json` contains the patch metadata that [!DNL CVT] uses to determine which patches apply to an installed version. [!DNL CVT] fetches the latest registry from Adobe's CDN and caches it under `var/patch_metadata/.patch_registry_cache.json`.

If the network is unavailable, [!DNL CVT] can use the cached registry with a warning. Use `--no-registry-cache` only when you require a fresh registry fetch.
