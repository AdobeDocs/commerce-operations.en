# ACSD-63687: Incorrect prices are displayed due to Redis cache cleanup issues

The ACSD-63687 patch fixes the issue where incorrect prices are displayed because it is not possible to clean Redis cache. This patch is available when the [[!DNL Quality Patches Tool (QPT)]](/help/tools/quality-patches-tool/quality-patches-tool-to-self-serve-quality-patches.md) 1.1.77 is installed. The patch ID is ACSD-63687. Please note that this issue is scheduled to be fixed in Adobe Commerce 2.4.9.

## Affected products and versions

**The patch is created for Adobe Commerce version:**

* Adobe Commerce (all deployment methods) 2.4.5-p9

**Compatible with Adobe Commerce versions:**

* Adobe Commerce (all deployment methods) 2.4.5 - 2.4.6-p13

>[!NOTE]
>
>The patch might become applicable to other versions with new [!DNL Quality Patches Tool] releases. To check if the patch is compatible with your Adobe Commerce version, update the `magento/quality-patches` package to the latest version and check the compatibility on the [[!DNL Quality Patches Tool]: Search for patches page](https://experienceleague.adobe.com/tools/commerce-quality-patches/index.html). Use the patch ID as a search keyword to locate the patch.

## Issue

Issue where incorrect prices are displayed because it is not possible to clean Redis cache.

<u>Steps to reproduce</u>:

1. Configure Redis.
1. Create multiple categories.
1. Create a script to send recurring request to a category page (Jmeter script could be used).
1. Perform block_html cache clean and run backend_clean_cache cron multiple times, while the requests are sent to the category page with script from previous step.
1. Execute script provided below to check the cache key records that do not exist in MAGE cache tag (Change the Redis port and the MAGE cache tag prefix according to the local project).

```
#!/bin/bash

REDIS_PORT=6379
REDIS_DB=0
SET_NAME="zc:ti:af6_MAGE"

redis-cli -p $REDIS_PORT -n $REDIS_DB keys "zc:k:*" | while read KEY; do
  STRIPPED_KEY=${KEY#"zc:k:"}
  EXISTS=$(redis-cli -p $REDIS_PORT -n $REDIS_DB sismember "$SET_NAME" "$STRIPPED_KEY")

  if [ "$EXISTS" -eq 0 ]; then
    TIMESTAMP=$(redis-cli -p $REDIS_PORT -n $REDIS_DB hget "$KEY" "m")

    if [ -n "$TIMESTAMP" ]; then
      FORMATTED_TIME=$(date -r "$TIMESTAMP" +"%y-%m-%d %H:%M:%S")
      echo "$KEY - m: $FORMATTED_TIME"
    else
      echo "$KEY - m: (null)"
    fi
  fi
done
```

<u>Expected results</u>:

Cache keys should not be created without the corresponding member records in cache tags.
backend_clean_cache cron should handle the cache tags cleanup operation correctly.

<u>Actual results</u>:

Cache keys are created without the corresponding member records in cache tags. As a result, cache clean by cache tags, triggered by backend_clean_cache cron, does not work.
