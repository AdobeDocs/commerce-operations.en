# ACP2E-4507: Password Options configuration does not apply to customer password reset requests made through GraphQL mutations

The ACP2E-4507 patch fixes the issue where Password Options configuration is not applied for customer password reset requests made through GraphQL mutations. This patch is available when the [[!DNL Quality Patches Tool (QPT)]](/help/tools/quality-patches-tool/quality-patches-tool-to-self-serve-quality-patches.md) 1.1.78 is installed. The patch ID is ACP2E-4507. Please note that this issue is scheduled to be fixed in Adobe Commerce 2.4.9.

## Affected products and versions

**The patch is created for Adobe Commerce version:**

* Adobe Commerce (all deployment methods) 2.4.8-p3

**Compatible with Adobe Commerce versions:**

* Adobe Commerce (all deployment methods) 2.4.8 - 2.4.8-p4

>[!NOTE]
>
>The patch might become applicable to other versions with new [!DNL Quality Patches Tool] releases. To check if the patch is compatible with your Adobe Commerce version, update the `magento/quality-patches` package to the latest version and check the compatibility on the [[!DNL Quality Patches Tool]: Search for patches page](https://experienceleague.adobe.com/tools/commerce-quality-patches/index.html). Use the patch ID as a search keyword to locate the patch.

## Issue

Issue where Password Options configuration is not applied for customer password reset requests made through GraphQL mutations.

<u>Steps to reproduce</u>:

1. Check and confirm the customer password options in
Store > Configurations > Customers > Customer Configuration > Password Options.
1. Send the password reset email using the requestPasswordResetEmail GraphQL mutation.
1. Reset the password using the resetPassword GraphQL mutation using the token from the sent email.
1. Repeat the same steps to reset the password multiple times.

<u>Expected results</u>:

1. Customer password resets triggered via GraphQL should behave the same way as web-based password resets and honor Admin configuration settings.

<u>Actual results</u>:

1. GraphQL currently allows triggering password reset requests an unlimited number of times and does not enforce the limits defined in the backend Customer Password Options.
